# 仓库维护约定

本文件用于约束后续对 `Code Review Agent` 项目的开发与维护方式。除非用户明确要求偏离，默认遵守以下规范。

## 1. 语言与表达

- 注释统一使用中文
- 文档统一使用中文
- README、示例、接口说明、错误提示优先使用中文
- 变量名、函数名、类名遵循 Python 社区习惯，保持英文命名
- 提交给用户的示例请求、示例响应、伪代码说明优先提供中文解释


## 2. 工程实践

- 使用 `uv` 管理项目依赖、虚拟环境与锁文件
- 新增依赖优先使用 `uv add`
- 新增开发依赖优先使用 `uv add --dev`
- 提交代码时应同时维护 `pyproject.toml` 与 `uv.lock`
- 使用 `src/` 布局组织 Python 包
- 优先补充类型标注，新增公开函数和方法应有明确类型签名
- 复杂逻辑需要写必要注释，但避免逐行翻译式注释，注释可以丰富一点，不要局限于一个方法一个注释

## 3. 架构约束

- 保持 `FastAPI + LangGraph` 为主骨架
- `API` 层只负责请求转换与响应输出，不直接承载复杂业务逻辑
- 评审主链路放在 `application` 与 `agent` 层
- `skills` 中的每个能力模块保持单一职责
- `tools` 负责外部工具或解析器封装
- `reporting` 负责 JSON / Markdown 输出
- 新功能优先以可扩展接口接入，避免把逻辑堆进单一大函数

## 4. 代码风格

- 遵循 PEP 8 和现代 Python 最佳实践
- 默认使用 `Pydantic v2`
- 默认使用显式类型标注
- 尽量使用不可变输入和小函数拆分提升可测试性
- 只在真正能提升理解时添加注释
- 避免无意义缩写，优先使用自解释命名

## 5. 质量保障

- 新增核心逻辑时，优先补充对应测试
- 至少运行：
  - `uv run ruff check .`
  - `uv run mypy src`
  - `uv run pytest`
- 如果因环境或外部依赖无法运行，需要在说明中明确指出

## 6. 文档维护

- 任何新增模块，若改变了项目结构或调用方式，应同步更新 README
- 若新增重要约束、开发规则或协作约定，应同步更新本文件
- 若新增 CLI 命令或 API 接口，应补充中文示例

## 7. 当前优先级

当前阶段优先目标是：

1. 保持最小可运行主链路稳定
2. 逐步接入真实静态分析工具
3. 为 RAG、审批、评估框架预留清晰扩展点
4. 保持代码、文档、注释风格统一，便于后续维护
